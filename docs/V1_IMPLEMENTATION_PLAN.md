# 供排水设施考核系统 V1 开发执行计划

## 1. V1 目标

V1 的目标不是先追求完整报告智能化，而是先把数据底座做扎实：

```text
移动端采集
-> 后端关系化入库
-> PC 看板汇总
-> PC 数据预览与复核
-> 数据锁定
-> 后续报告生成和 Agent 增强读取正式数据
```

当前阶段以数据库为边界：前三步先保证数据可信、可复核、可追溯；报告生成和 Agent 放在正式数据稳定之后增强。

## 2. 当前真实目录

```text
Agent/
  backend/                 FastAPI 后端
  frontend/
    front/                 PC 平台端
    front-mobile/          移动端采集端
  tests/                   联调和验证脚本

docs/
  BACKEND_DEVELOPMENT_PLAN.md
  V1_IMPLEMENTATION_PLAN.md
```

## 3. 技术栈

- 后端：FastAPI、SQLAlchemy、Alembic、Pydantic
- 数据库：PostgreSQL 为目标，本地开发可使用 SQLite
- 文件：本地文件存储起步，后续可迁移 MinIO
- 异步任务：Redis + Celery，开发期可先同步执行
- 前端：React + Vite + Tailwind
- 报告：python-docx 等 Python 文档生成工具
- Agent/大模型：后续作为增强模块，不直接决定分数、金额和锁定状态

## 4. 开发总顺序

优先级固定为：

1. 后端补齐移动端入库接口
2. 评分标准配置完整化
3. 移动端对接真实接口和离线暂存
4. PC 端接入真实数据、看板和复核
5. 报告生成接正式数据
6. Agent/大模型增强

其中 V1 近期重点是第 1 到第 4 步。

## 5. 阶段 1：后端补齐移动端入库接口

目标：

- 创建考核主记录时绑定 `city_id`、`cycle_id`、`town_id`、`village_id`、`indicator_version_id`。
- `raw_payload` 保留为原始提交快照，但不能作为长期唯一数据源。
- 评分写入 `assessment_scores`。
- 问卷写入 `survey_records`。
- 水质写入 `water_quality_records`。
- 照片写入 `attachments`，并支持绑定 `score_id` 和 `deduction_option_id`。

验收：

- 一个村点的评分、问卷、水质、照片能保存到对应关系表。
- 移动端提交后，PC 能读取同一条记录详情。
- 锁定后，移动端不能再次提交、改分、补传照片。
- 后端 `/health` 正常。

当前状态：

- 主链路已经基本打通。
- 仍需继续补数据唯一性、防重复提交、状态流转边界测试。

## 6. 阶段 1.5：评分标准配置完整化

这是前三步的数据地基，必须在报告和 Agent 之前完成。

当前问题：

- 后端现在只有最小示例标准：`产出 -> 项目运营 -> 污水收集`，满分 20 分。
- 评分标准版本表和复制接口已经有骨架，但完整 100 分标准尚未入库。
- 移动端目前还没有完整的评分标准版本选择界面。

目标：

- 将完整评分标准导入数据库，而不是继续写死在移动端。
- 覆盖一级指标、二级指标、三级指标、满分、评价标准、扣分项、是否需要照片、是否来源问卷、是否仅作水质佐证。
- 评分标准以版本管理，不直接覆盖历史版本。
- 移动端根据城市、批次、设施类型加载对应已发布评分标准版本。
- 每条考核记录必须绑定 `indicator_version_id`。

核心表：

- `indicator_versions`：评分标准版本。
- `indicators`：一级、二级、三级指标树。
- `deduction_options`：三级指标下的评价标准和扣分项。
- `score_source_mappings`：问卷、水质等来源数据与正式评分指标的关系。

版本规则：

- 标准版本状态至少包括 `draft`、`published`、`locked`。
- 草稿可编辑。
- 发布后移动端可读取。
- 已被考核记录使用或已生成报告的版本应锁定。
- 城市标准微调时，走“复制旧版本 -> 微调 -> 发布 -> 锁定历史版本”。

移动端规则：

- 城市选择作为第一步固定保留。
- 城市确定后选择考核批次。
- 后端根据城市、批次、设施类型返回当前发布版评分标准。
- 移动端原则上不让采集人员选择草稿版本。

验收：

- 移动端看到的不再是一个 20 分示例，而是完整指标树。
- 三级指标能展示对应评价标准和扣分项。
- 问卷来源指标不开放人工填写，只显示等待回填或已回填。
- 水质抽检只作为佐证，不直接扣分。
- PC 复核页能显示三级指标名称、扣分项名称和标准版本。

## 7. 阶段 2：移动端对接真实接口

目标：

- 保持现有移动端页面体验。
- 提交时传城市、批次、镇街、村点、标准版本、评分、问卷、水质、照片绑定信息。
- 页面刷新后能恢复草稿或已提交记录。
- 增加离线暂存能力，联网后同步后端。

离线暂存建议：

- 本地保存草稿和同步队列。
- 每条本地记录包含 `local_id`、`sync_status`、`last_error`、`synced_at`。
- 状态至少包括 `draft`、`pending_sync`、`synced`、`sync_failed`。
- 同步成功后写入后端返回的 `record_id`。

验收：

- 移动端断网时能继续填表和保存草稿。
- 联网后能补传数据。
- 同一城市、批次、镇街、村点、设施类型不应重复生成多条主记录。
- 退回后员工能继续修改并重新提交。
- 锁定后移动端只读。

当前状态：

- 城市、登录、提交后端、评分、问卷、水质入库已经接入。
- 本地暂存已有基础能力。
- 离线同步队列、防重复提交、退回后再提交还需要继续细化。

## 8. 阶段 3：PC 看板和数据复核

目标：

- PC 看板读取后端汇总。
- 支持查看镇街完成率、村点进度、扣分统计、缺照片、低分、待复核、已退回等信息。
- 数据预览页支持查看、编辑、退回、复核、锁定移动端提交记录。
- 所有 PC 人工修改写入复核日志。

看板能力：

- 镇街进度
- 完成率
- 待复核记录
- 低分记录
- 缺照片记录
- 已退回记录
- 扣分统计

复核能力：

- 按镇街、状态、风险筛选。
- 查看城市、批次、镇街、村点、评分标准版本。
- 查看评分明细、三级指标名称、扣分项名称、满分、得分、扣分说明。
- 查看问卷、水质、照片证据。
- 修改评分时必须填写调整原因。
- 退回时填写退回原因。
- 复核、退回、锁定写入 `review_logs`。
- 锁定后所有编辑接口拒绝修改。

验收：

- PC 能筛出待复核、低分、缺照片记录。
- PC 能从看板跳转到指定镇街复核记录。
- PC 能修改评分，并记录修改前后值、修改人、原因、时间。
- PC 能退回、复核、锁定。
- 锁定后移动端和 PC 都不能再改。

当前状态：

- 看板真实数据、预警统计、看板跳转复核页已经有基础版。
- 复核页已支持状态/风险筛选、评分编辑、调整原因、退回、锁定、日志。
- 仍需继续补更完整的数据唯一性、修改前后值展示、退回后移动端再提交闭环。

## 9. 阶段 4：报告生成接正式数据

该阶段在前三步稳定后再重点推进。

启动前提：

- 阶段 1 到阶段 3 已跑通至少一条完整记录。
- 评分、问卷、水质、照片均已关系化入库。
- PC 端可以完成复核、退回、锁定。
- 评分标准版本已经绑定到考核记录。

目标：

- 报告任务从已复核或已锁定数据生成。
- 报告中引用评分、问卷、水质、照片、复核结论。
- 生成结果保存为报告文件记录，可下载和追溯。

原则：

- 报告读取关系表和锁定数据，不直接读取移动端临时 JSON。
- 分数、扣分、金额、锁定状态不由大模型决定。
- Agent 只用于资料理解、问题归纳、报告段落生成和语义校验。

详细任务：

1. 报告数据集构建
   - 建立报告生成专用查询服务，例如 `report_dataset`。
   - 输入为城市、批次、镇街、村点、报告类型。
   - 输出结构化数据，包括评分明细、扣分项、问卷、照片、水质、复核日志和基础信息。
   - 报告数据集只读取已复核或已锁定记录。
   - 数据集生成时固化 `record_id`、`indicator_version_id`、`cycle_id` 和数据快照摘要。

2. 报告模板与数据映射
   - 梳理现有 DOCX 报告模板章节。
   - 将章节所需数据映射到数据库字段。
   - 明确哪些章节来自确定性数据，哪些章节可由 Agent 辅助生成。
   - 模板中金额、分数、扣分、排名等必须来自后端确定性计算。
   - 建立模板字段清单，至少包括基本信息、总分、一级/二级/三级得分、主要扣分项、照片证据、水质佐证、问卷结果、复核结论。
   - 模板字段缺失时，报告任务应失败并返回明确错误，不生成半成品正式报告。

3. 报告任务系统
   - `report_tasks` 记录任务状态、输入参数、进度、错误信息和创建人。
   - `reports` 记录生成文件、所属批次、镇街、文件大小、文件状态和下载地址。
   - 支持任务状态：`queued`、`running`、`completed`、`failed`。
   - 前端报告生成页轮询任务状态。
   - 开发期可同步生成，生产期再切换 Celery/Redis 异步执行。
   - 任务失败时保留错误堆栈摘要和输入参数，便于排障。

4. 报告输出类型
   - 单镇街报告。
   - 多镇街汇总报告。
   - 附件清单或证据目录。
   - 后续可扩展为 PDF 导出。
   - V1 优先完成单镇街 DOCX，再做批次汇总和 PDF。

5. 报告可追溯
   - 每份报告记录使用的数据范围。
   - 记录报告生成时绑定的评分标准版本。
   - 记录生成时间、操作人、任务参数。
   - 锁定数据发生变更时，禁止覆盖旧报告，只能重新生成新报告版本。

6. PC 报告页面
   - 新增报告任务列表。
   - 支持按城市、批次、镇街、状态筛选。
   - 支持创建报告、查看任务进度、下载报告、查看失败原因。
   - 报告生成前提示数据范围和锁定状态。

验收：

- PC 端只能基于已复核或已锁定数据创建报告任务。
- 报告生成失败时能在前端看到明确错误。
- 报告能下载。
- 报告正文至少能引用镇街、村点、总分、扣分项、问卷统计、水质记录和照片数量。
- 报告生成结果能在历史报告中追溯。

## 10. 阶段 5：Agent/大模型增强

目标：

- 资料理解。
- 问题描述归纳。
- 报告段落生成。
- 报告语义校验。
- 异常提示和辅助复核。

边界：

- 大模型不直接决定最终评分。
- 大模型不直接决定金额。
- 大模型不绕过数据锁定状态。
- Agent 输出必须带来源依据或结构化结果。

详细任务：

1. Agent 能力边界
   - 资料理解：读取合同、补充协议、绩效说明、历史报告等非结构化资料。
   - 问题归纳：根据扣分项、照片说明、复核意见生成问题描述。
   - 报告段落生成：生成管理建议、整改建议、问题分析段落。
   - 语义校验：检查报告前后表述是否矛盾。
   - 不参与最终分数、金额和锁定状态判断。

2. Agent 服务边界
   - 业务后端继续负责登录、权限、批次、评分、复核、锁定和报告任务。
   - Agent 模块只作为能力服务，被报告任务或复核页调用。
   - Agent 调用应通过后端封装，不让前端直接调用大模型 API。
   - 后续如果切换 Spring Boot，Agent 模块仍可保留为独立 Python 服务。

3. Agent 输入数据
   - 结构化评分数据来自数据库。
   - 非结构化资料来自上传文件和解析结果。
   - 报告上下文来自报告数据集。
   - 所有 Agent 输入应带来源 ID 或文件引用。
   - 输入给大模型前应脱敏处理员工账号、手机号等不必要信息。

4. Agent 输出格式
   - 输出 JSON 或结构化字段，不直接写入最终报告正文。
   - 输出应包含 `summary`、`evidence_refs`、`confidence`、`warnings` 等字段。
   - 低置信度内容进入人工复核，不自动入报告。
   - 输出字段需要通过 Pydantic 校验，校验失败则回退到基础报告。

5. 模型适配层
   - 保留模型供应商适配层，避免业务代码直接绑定某一个模型。
   - OpenAI Agents SDK 可用于工具调用、流程编排、日志追踪。
   - 其他大模型 API 可通过适配器接入，但必须验证工具调用、结构化输出和上下文能力。
   - 模型配置至少预留 `provider`、`model`、`base_url`、`api_key_ref`、`enabled`。

6. 人工确认流程
   - Agent 生成的段落应进入 PC 端预览。
   - 管理员确认后再写入报告任务。
   - Agent 生成内容、人工确认人、确认时间应可追溯。

7. Agent 质量控制
   - 对照数据库总分、扣分项和水质记录做一致性检查。
   - 禁止生成与锁定数据相矛盾的分数、排名和金额。
   - 对缺少来源依据的句子标记为待确认。
   - 记录每次 Agent 调用的输入摘要、输出摘要、耗时和错误。

验收：

- Agent 失败时，系统仍能生成基础报告。
- Agent 输出不会覆盖数据库中的分数和金额。
- Agent 生成段落能看到来源依据。
- 管理员能在 PC 端确认或弃用 Agent 结果。
- 后端日志能追溯一次报告生成中使用了哪些 Agent 能力。

## 11. 阶段 6：部署、权限和运维

该阶段用于从本地原型走向内部员工可用系统。

目标：

- 支持内部员工通过浏览器使用 PC 端和移动端。
- 后端、前端、数据库和文件存储能够稳定部署。
- 具备基础账号、权限、备份和日志能力。

详细任务：

1. 部署形态
   - PC 端和移动端作为两个 H5 前端部署。
   - 后端 FastAPI 作为 API 服务部署。
   - PostgreSQL 作为正式数据库。
   - 文件存储使用本地挂载目录或 MinIO。
   - Nginx 统一反向代理。
   - 建议路径：`/platform` 指向 PC 端，`/mobile` 指向移动端，`/api` 指向后端。
   - 开发端口继续保留：后端 `8000`，PC `5173`，移动端 `5174`。

2. 访问方式
   - 员工手机直接访问移动端 H5 地址。
   - 管理员电脑访问 PC 平台地址。
   - V1 不强制包装原生 App。
   - 如果后续需要类似 App 的体验，可优先做 PWA，再评估是否封装原生 App。

3. 账号权限
   - 普通用户：移动端采集、查看本人任务、修改退回记录。
   - 管理员：批次管理、标准管理、复核、锁定、报告生成。
   - 后续可扩展项目负责人、只读审阅人等角色。
   - V1 至少完成登录、token 有效期、接口鉴权和角色校验。

4. 数据安全
   - 生产环境不使用开发 token。
   - 登录密码或企业微信登录后续补充。
   - 文件上传限制类型和大小。
   - 附件下载需要鉴权。
   - 生产环境配置文件不得提交真实密钥。

5. 数据备份
   - PostgreSQL 定期备份。
   - 文件存储定期备份。
   - 报告文件和原始附件保留版本。
   - 备份恢复流程至少在测试环境演练一次。

6. 运行监控
   - 后端错误日志。
   - 报告任务失败日志。
   - Agent 调用日志。
   - 文件上传和下载日志。

7. 环境划分
   - `dev`：本地开发，SQLite 或本地 PostgreSQL。
   - `test`：联调测试，使用接近生产的 PostgreSQL 和文件存储。
   - `prod`：正式使用，启用正式登录、备份、日志和反向代理。

8. 启停脚本
   - Mac 保留 `start-mac.sh`，用于本地同时启动后端、PC 前端、移动端。
   - Windows 可保留对应启动脚本，但脚本命名使用英文，避免目录和脚本名称混乱。
   - 生产环境不依赖本地开发脚本，改用 systemd、Docker Compose 或云服务进程管理。

验收：

- 内网或云服务器部署后，员工手机可访问移动端。
- 管理员可访问 PC 平台。
- 数据库和文件存储重启后数据不丢失。
- 报告任务失败时能定位错误。
- 普通用户不能访问管理员接口。
- 生产环境重启后，服务能自动恢复或按标准流程恢复。

## 12. 阶段 7：正式试运行和业务验收

目标：

- 选择一个城市、一个批次、少量镇街做试运行。
- 验证采集、复核、锁定、报告生成完整闭环。

试运行流程：

1. 管理员创建城市、批次和评分标准版本。
2. 移动端采集 1 到 2 个镇街的村点数据。
3. PC 看板检查进度、低分、缺照片、待复核。
4. 管理员复核、退回、重新提交、锁定。
5. 基于锁定数据生成报告。
6. 对报告内容和数据库记录进行抽查。

试运行数据范围：

- 至少 1 个城市。
- 至少 1 个考核批次。
- 至少 2 个镇街。
- 每个镇街至少 2 个村点或设施样本。
- 至少覆盖设施类和管网类两套评分标准。

角色参与：

- 采集人员：完成移动端录入、照片上传、退回修改。
- 管理员：完成标准发布、复核、退回、锁定、报告生成。
- 业务审核人：抽查报告内容和佐证材料。

问题闭环：

- 试运行问题分为数据问题、界面问题、规则问题、报告问题、Agent 问题。
- 数据和规则问题优先修复。
- 报告和 Agent 问题在数据闭环稳定后迭代。
- 每轮试运行结束形成问题清单和下一轮修订计划。

验收：

- 采集人员能独立完成移动端填报。
- 管理员能完成复核和锁定。
- 系统能生成可用报告初稿。
- 数据库记录和报告内容能够互相追溯。
- 发现的问题能回填到下一轮开发清单。

## 13. 阶段 8：后续扩展和迁移预留

该阶段不作为 V1 必须完成项，但要在前面开发时预留边界，避免后期大返工。

可能扩展：

- 多城市标准微调和复制发布。
- 多项目、多批次并行管理。
- 企业微信或统一身份认证登录。
- 原生 App 或 PWA 离线能力增强。
- MinIO、对象存储、云数据库等生产级基础设施。
- Spring Boot 平台后端迁移。
- 更多报告模板和城市定制版模板。
- 更多 Agent 能力，例如资料自动抽取、整改建议对比、历史批次趋势分析。

迁移原则：

- 数据库表结构和 API 合约保持稳定。
- 评分标准、评分记录、复核日志、报告任务不要依赖前端 mock 数据。
- Agent 能力通过服务接口调用，不写进评分核心逻辑。
- 若后续迁移 Spring Boot，优先迁移业务 API 和权限，Python 继续保留报告生成和 Agent 能力。
- 前端只依赖 HTTP API，不直接依赖后端语言。

不在 V1 立即做：

- 复杂多租户体系。
- 原生 App 上架。
- 全量政企级权限矩阵。
- 全自动无人审核报告。
- 大模型自动判分。

## 14. V1 验收测试清单

前三步验收：

- 后端 `/health` 正常返回。
- 移动端可完成城市、批次、镇街、村点、评分标准选择。
- 移动端能加载完整评分标准版本，而不是 20 分示例。
- 一个村点的评分、问卷、水质、照片能保存到对应关系表。
- 页面刷新后能恢复草稿或已提交记录。
- 离线暂存、联网同步能跑通。
- 问卷来源指标能自动回填并进入总分。
- 水质记录不影响扣分，但能在 PC 复核和报告中作为佐证出现。
- 照片能绑定到具体评分明细或扣分项。
- PC 看板能看到移动端提交后的进度和扣分统计。
- PC 看板能看到待复核、低分、缺照片、退回记录。
- PC 复核修改有日志。
- 锁定后普通编辑接口不能再改。
- 数据库能查出一条完整记录的所有来源数据。

后续报告验收：

- 报告任务能基于已复核或已锁定数据生成 DOCX。
- 报告文件能下载和追溯。
- 报告内容能引用评分、问卷、水质、照片和复核结论。

Agent 和部署验收：

- Agent 失败不影响基础报告生成。
- Agent 生成内容需要管理员确认。
- 生产环境使用正式数据库和持久化文件存储。
- 员工手机可以访问移动端 H5。
- 管理员电脑可以访问 PC 平台。

## 15. 当前下一步建议

下一步优先做：

1. 将完整评分标准导入数据库。
2. 移动端加载完整标准版本，替换 20 分示例。
3. 增加移动端标准版本展示，明确当前城市、批次、标准版本。
4. 补记录防重复提交和离线同步队列。

这样前三步的数据地基会比继续做报告更稳。
